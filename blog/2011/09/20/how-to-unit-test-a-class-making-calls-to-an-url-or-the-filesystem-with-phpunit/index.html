How to Unit Test a Class Making Calls to an URL (or the Filesystem) With PHPUnit

Sep 20th, 2011

For our most recent After Work Hacking my co-workers and me decided to write unit tests forthe open source project Assetic. That turned out to be a better decision than our last one, yet we still ran into an interesting challenge.

Ok, nice, so is there anything else in the constructor we could write a test for?
No, there isn’t anything left in the constructor we can directly test. In order to test the rest of the code of the constructor, we have to call the load()-method.

So here’s the rather easy test of passing an URL into HttpAsset which passes the tests in the constructor but isn’t actually a valid URL and thus causes load() to throw an exception:

But now it’s getting interesting:

The problem is in line 42 of HttpAsset where the native PHP function file_get_contents() is called. It isn’t actually a call to the filesystem - file_get_contents can also handle URLs - but how can we test that?

Here are the possibilities we have at our disposal:

1. Use a real URL

The obvious thing to do is simply pass a real URL into HttpAsset. That’s also the option chosen by the authors of the tests for HttpAsset (there already are tests for Assetic but we threw them away to practice):

However, this method has a number of problems:

  1. The test will fail when there’s no connection to the internet.
  2. The test will fail when the URL you use changes.
  3. You have little control about the content returned by the call to the URL.

Ok, what else could we do? Read on!

2. Wrap the use of file_get_contents() into a new class.

If you move the code using the filesystem or another external resource into a new class and inject it into HttpAsset, you will able to mock that class when testing HttpAsset.
Normally, that would be my preferred way of making classes like HttpAsset easier to test.

However, in this case I’m not so sure about it.
To see why, let’s check out how HttpAsset would look like if we moved all code concerned with getting the content over HTTP into a new class ContentFetcher:

I’ll leave the implementation of ContentFetcher as an exercise for the reader (yes, I can be a lazy bum sometimes… ;) ). Note how I don’t pass an URL into HttpAsset anymore but an instance of ContentFetcher. Accordingly, all code responsible for validating the URL has been moved into ContentFetcher.
Anyway, now it isn’t any problem at all anymore to write some tests avoiding the problems mentioned in the previous section, for you’re able to mock ContentFetcher with PHPUnit.
Additionally, HttpAsset has become much simpler.

Well, I think it has maybe become too simple, there really isn’t anything interesting going on anymore in HttpAsset and that’s because HttpAsset itself already was a class to abstract away from the nitty-gritty details of fetching or loading content via HTTP.

Hence, in this particular case, it’s probably better to leave HttpAsset as it is and look to another option.

3. Mock the filesystem with vfsStream

This sounds suspiciously similar to the previous option - but it isn’t!
What I mean is to mock the underlying filesystem, i.e. replace the “real” filesystem with a virtual filesystem, thus allowing you to have full control about what get returned etc.

And fortunately, a library making this possible exists: vfsStream.
vfsStream is a stream wrapper for a virtual filesystem allowing you to mock the real filesystem.
This allows you to possibly keep the code under test unchanged, you simply swap out the real filesystem with a fake one you have full control over.

Explaining how to use vfsStream is out of the scope of this blog but luckily, you can find some examples in the documentation for PHPUnit:
Using vfsStream with PHPUnit

The bottom line is, whenever you need to test a class accessing the filesystem, e.g. removing or creating directories, take a look at vfsStream, it will be of a great help!

Yet, in our particular case vfsStream cannot be used.

First off, it doesn’t support requests via HTTP, as it’s done in HttpAsset using file_get_contents(). That’s the bad news, the good news is: After having skimmed the source code of vfsStream, it looks like it is possible to add such a feature to vfsStream!

Alas, secondly, you can only use vfsStream, if the code under test doesn’t itself create a stream wrapper - and that’s what HttpAsset is doing in getLastModified.

What a bummer. ;)

Conclusion

In the case of testing HttpAsset, I don’t see any way how we could get around the test having to make a request to a real URL - do you?

However, in other cases, the options I’ve explored in this blog might be a valid alternative. So the next time you have to test a class using the filesystem or performing HTTP requests:

  • try to wrap the offending code into a new class, so you can mock it, or
  • try to mock the filesystem with vfsStream.

Hope that helps!
